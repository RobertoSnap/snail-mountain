# Snail Mountain вҖ” Architecture

## Overview

**Snail Mountain** is a SvelteKit 5 web application scaffolded with a full-stack TypeScript setup targeting Node.js server deployment. It includes authentication (Better Auth), a SQLite database (Drizzle ORM + LibSQL), Tailwind CSS v4 styling, and a comprehensive testing stack (Vitest + Playwright).

| Aspect | Technology |
|--------|-----------|
| Framework | SvelteKit 2 (Svelte 5) |
| Language | TypeScript (strict mode) |
| Package Manager | pnpm |
| Runtime Adapter | @sveltejs/adapter-node |
| Database | SQLite via LibSQL + Drizzle ORM |
| Authentication | better-auth (email/password) |
| Styling | Tailwind CSS v4 + forms/typography plugins |
| Unit/Component Tests | Vitest (node + browser via Playwright) |
| E2E Tests | Playwright |
| Linting | ESLint (TS + Svelte + Prettier compat) |
| Formatting | Prettier (svelte + tailwind plugins) |
| Build Tool | Vite 7 |

### Authentication Layer

Authentication is managed entirely server-side using **better-auth**:

- **Configuration** (`src/lib/server/auth.ts`): Initializes better-auth with the Drizzle adapter (SQLite), email/password strategy, and SvelteKit cookie integration via `sveltekitCookies(getRequestEvent)`.
- **Session Middleware** (`src/hooks.server.ts`): Every incoming request passes through the `handleBetterAuth` hook, which resolves the session from request headers and populates `event.locals.user` and `event.locals.session`.
- **Type Safety** (`src/app.d.ts`): The `App.Locals` interface is augmented with optional `User` and `Session` types from better-auth.
- **Route Guards**: Individual `+page.server.ts` load functions check `event.locals.user` and redirect unauthenticated users.
- **Form Actions**: Sign-in, sign-up, and sign-out are implemented as SvelteKit form actions with progressive enhancement (`use:enhance`).

### Environment Validation

- **Centralized validation** (`src/lib/server/env.ts`): All required environment variables are validated at startup using Zod schemas.
- **Validated variables**: `DATABASE_URL`, `ORIGIN`, `BETTER_AUTH_SECRET` вҖ” all must be non-empty strings.
- **Fail-fast**: The app throws a clear error listing any missing variables before the server starts.
- **Single import**: All server modules import from `$lib/server/env` instead of `$env/dynamic/private` directly.

### Database Layer

- **Client**: LibSQL client (`@libsql/client`) configured via `DATABASE_URL` environment variable.
- **ORM**: Drizzle ORM wraps the LibSQL client with typed schema access.
- **Schema** (`src/lib/server/db/schema.ts`):
  - `task` table: `id` (UUID primary key), `title` (text, not null), `priority` (integer, default 1)
  - Re-exports auth schema from `auth.schema.ts` (currently a placeholder)
- **Configuration** (`drizzle.config.ts`): SQLite dialect, verbose and strict mode enabled.
- **Scripts**: `db:push`, `db:generate`, `db:migrate`, `db:studio` for database management.

### Styling

- **Tailwind CSS v4** integrated via `@tailwindcss/vite` plugin.
- **Plugins**: `@tailwindcss/forms` and `@tailwindcss/typography` loaded in `layout.css`.
- **Prettier Integration**: `prettier-plugin-tailwindcss` auto-sorts utility classes with the stylesheet path configured to `./src/routes/layout.css`.

### Testing Architecture

The project uses a **dual test runner** setup:

| Layer | Runner | Environment | Pattern |
|-------|--------|-------------|---------|
| Unit tests | Vitest | Node.js | `src/**/*.{test,spec}.{js,ts}` (excluding `.svelte.*`) |
| Component tests | Vitest + Playwright | Browser (Chromium) | `src/**/*.svelte.{test,spec}.{js,ts}` |
| E2E tests | Playwright | Full app (build + preview) | `e2e/**/*.test.ts` |

- **Assertion requirement**: `expect.requireAssertions` is enabled globally.
- **Server tests excluded from browser**: `src/lib/server/**` is excluded from browser test project.

## Improvement Recommendations

### рҹҸ—пёҸ Architecture & Structure

1. вң… **Create a shared components library** (Done): No `src/lib/components/` directory exists. Extract reusable UI elements (buttons, form inputs, cards, alerts) to reduce duplication and enforce consistency as the app grows.

2. вң… **Add error boundaries** (Done): No `+error.svelte` pages exist at any route level. Add at minimum a root-level error page, and consider route-group-specific error pages for tailored error UX.

3. вң… **Add route group layouts** (Done): The `demo/` section and especially `demo/better-auth/` would benefit from their own `+layout.svelte` files for shared navigation, breadcrumbs, or page chrome.

4. **Populate or remove `lib/index.ts`**: The barrel export is empty. Either use it for shared utilities/types or remove it to avoid confusion.

5. вң… **Generate the auth schema** (Done): `auth.schema.ts` is still a placeholder comment. Run `pnpm auth:schema` to generate the actual Better Auth database tables вҖ” the app cannot properly persist sessions until this is done.

### рҹ”’ Security

6. вң… **Add input validation** (Done): Form inputs are extracted raw from FormData with only `toString()` coercion. Adopt a validation library (e.g., Zod) and consider SvelteKit Superforms for type-safe, validated form handling.

7. **Add rate limiting**: Login and registration endpoints have no rate limiting, leaving them vulnerable to brute-force attacks. Consider middleware-level rate limiting or a better-auth plugin.

8. вң… **Validate all environment variables at startup** (Done): Only `DATABASE_URL` is validated with a thrown error. `BETTER_AUTH_SECRET` and `ORIGIN` are unvalidated and will cause silent/confusing failures if missing. Add a centralized env validation module (e.g., using Zod or `$env/dynamic/private` with explicit checks).

9. вң… **Remove dead callback URL** (Done): The `callbackURL: '/auth/verification-success'` in login actions references a route that does not exist in the codebase. Either create this route or remove the callback.

### рҹ§Ә Testing

10. **Expand test coverage significantly**: Current tests are scaffolding-only (1 arithmetic unit test, 1 h1 check component test, 1 h1 check E2E test). Priority areas for new tests:
    - Auth flow integration tests (sign-up, sign-in, sign-out, guard redirects)
    - Database operations (task CRUD)
    - Form validation error paths
    - Server load function behavior

11. вң… **Fix test file placement** (Done): `src/demo.spec.ts` is a server-side unit test sitting at the src root. Move it into a structured test directory or colocate with the module it tests.

12. **Add test utilities**: Create shared test helpers for database seeding, authentication mocking, and request factories.

### рҹҡҖ DevOps & Deployment

13. вң… **Add a CI/CD pipeline** (Done): No automation exists. Set up GitHub Actions (or equivalent) with jobs for: linting, type-checking, unit tests, E2E tests, and deployment.

14. вң… **Add Docker support** (Done): Since the project uses `adapter-node`, a `Dockerfile` and optionally `docker-compose.yml` would streamline deployment and local environment parity.

15. **Establish migration workflow**: The `drizzle/` directory is referenced in `.prettierignore` but no migrations exist. Move from `db:push` (development-only) to a proper `db:generate` вҶ’ `db:migrate` workflow before production.

### рҹҺЁ UI & UX

16. вң… **Add global navigation** (Done): There is no persistent navigation component. Users cannot discover or move between sections of the app.

17. вң… **Extract duplicated styles** (Done): The login form repeats identical Tailwind class strings across multiple input elements. Extract a shared `<Input>` component or use Tailwind `@apply` for form element base styles.

18. **Add loading indicators**: No loading UI exists for server-loaded pages. SvelteKit's `{#await}` blocks or navigation loading indicators would improve perceived performance.

19. **Add a favicon reference in app.html**: While the layout sets a favicon via Svelte component, adding a static fallback in `app.html` ensures the icon appears even before JS hydration.
